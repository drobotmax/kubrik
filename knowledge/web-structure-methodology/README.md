# Web Structure Methodology (UPH reference case)

База знаний по эталонному процессу разработки B2B-сайта "с нуля": паспорт проекта -> структура -> прототип -> ТЗ -> разработка. Основана на реальном кейсе UPH (производитель аналитического и досмотрового оборудования).

## Когда обращаться к этой базе

Claude должен читать и применять эту методологию в случаях:
- Клиент просит **разработку сайта/лендинга/корп. сайта/каталога** с нуля
- Нужно спроектировать **структуру B2B сайта** (каталог без покупки, отраслевые решения, госзакупки, дилеры, сотрудничество)
- Нужен **чеклист блоков** для главной / под-категории / карточки товара / отраслевой страницы
- Запрос на **паспорт проекта** (бриф с целями, ЦА, CJM)
- Клиент с **B2B / госсектором / производством** - здесь особая логика: госзакупки, дилеры, КБ, вузы
- Нужно понять, **какие вопросы задавать клиенту** на созвоне по структуре (фиксация "пустот" на доработку)
- Ценообразование на **прототип + ТЗ** для референса (для KUBRIK - не обязательная константа, но ориентир)

## Ключевые инсайты кейса UPH

1. **3 варианта группировки каталога** (технология / отрасль / вид) - выбор зависит от ЦРМ/1С клиента; без сверки с ERP каталог придётся переделывать.
2. **Бургер-меню каталога** как современный SEO-паттерн для большой номенклатуры.
3. **Пустая поисковая выдача -> форма обратной связи**, а не "ничего не найдено". Не отпускаем заинтересованного пользователя.
4. **Отраслевые страницы с интерактивной схемой предприятия** (входная группа / лаборатория) + кликабельные карточки оборудования в схеме.
5. **Госзакупки = отдельная страница** (B2G-трек), если клиент работает с 44-ФЗ/223-ФЗ.
6. **Дилеры = отдельная страница + отдельный процесс**: спросить у клиента, как работает входящий лид дилера (анкета vs форма), нужно ли показывать существующих дилеров на карте (часто нет - "рынок захвачен" демотивирует новых).
7. **Сотрудничество разделяется**: научно-конструкторские бюро + вузы. У каждого своя логика, форма, отзывы.
8. **Сервис после гарантии** = отдельный блок удержания клиентов (не теряем их после окончания гарантийного периода).
9. **Формы обратной связи**: выпадашка "отрасль" нужна только если у клиента в ОП менеджеры разделены по отраслям.
10. **Price на сайте** - только если в компании есть кто-то, кто следит за актуальностью. Иначе лучше не размещать.

## Паттерн работы проектировщика (на кейсе UPH)

1. **Паспорт** (созвон 1) - цели проекта, ЦА, позиционирование, конкуренты.
2. **Структура** - проектировщик делает Miro со структурой + несколько вариантов для спорных моментов (каталог 3 способами группировки).
3. **Внутренняя проверка** - созвон с коллегой (мини-walkthrough, чтобы поймать проблемы до клиента).
4. **Демо клиенту** - пошаговый разбор каждой страницы; в реальном времени фиксируются "пустоты" (что нужно уточнить у клиента).
5. **Финализация структуры** на обсуждениях.
6. **Прототип** (Excalidraw / Figma wireframes).
7. **Техническое задание**.
8. **Дизайн + разработка**.

## Ценовой ориентир (UPH, 2026-03)

- Прототип: **40-60К руб** (после согласования структуры - в середине вилки)
- ТЗ: **30-40К руб**

Это референс, не KUBRIK-стандарт. KUBRIK-pricing определяется отдельно исходя из сложности и позиционирования.
